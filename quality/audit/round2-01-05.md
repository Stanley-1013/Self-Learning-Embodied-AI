## Ch03 C++ Memory Model & Atomics

### New findings
- **[HIGH]** `03-cpp-memory-model-atomics.md:497, 513, 523` — Three residual stale claims that directly contradict the round-1 ARMv8 mapping rewrite in the main text (lines 49, 55, 112-119, 430-438):
  - Line 497 (Q3): "3 sensor × 1 kHz = 3000 次 acquire load/s ... **若改 seq_cst 需 `DMB` ≈ 50–100 cycles → 每秒浪費 15–30 萬 cycles**". The whole point of the main-text fix is that a pure seq_cst **load** on ARMv8 is still `LDAR`, identical to acquire load — no DMB — so the "15–30 萬 cycles 浪費" number is zero for this scenario. This is exactly the "30 萬 cycles" false-premise calculation the round-1 audit flagged, and the fix only removed it from one place.
  - Line 513 (Q4): "ARM 上 ... seq_cst 是 `STLR + DMB`". Wrong for pure store — it's just `STLR`. Line 112 in the same file explicitly contradicts this.
  - Line 523 (面試角度): "`seq_cst` 的 DMB 全屏障代價比 x86 大一個數量級" — framed as if every seq_cst op has a DMB, not just the RMW path.
  - Suggested fix: rewrite each bullet so the cost argument keys off seq_cst **RMW** (fetch_add/exchange/CAS) and compiler cross-variable reorder restrictions — not a phantom DMB on plain load/store. Q3's "浪費 X cycles" quantitative example should be moved to a CAS/fetch_add scenario, or the scenario recast as pure pointer swap with the honest answer "LDAR vs LDAR → 0 差異 on ARMv8; acquire-release 的收益其實在 RMW 與 compiler 排程".
  - Confidence: HIGH. This is a classic incomplete-fix pattern: the principle is corrected but the quantitative hooks that originally sold it are still floating around.

- **[LOW]** `03-cpp-memory-model-atomics.md:59` — "`std::atomic<T>` 透過硬體指令（`lock cmpxchg` on x86、`ldxr/stxr` on ARM）實現原子性". `lock cmpxchg` / `ldxr/stxr` are CAS/LL-SC primitives, not how atomic **load/store** is realized (plain `mov` / `ldr`/`str` is atomic for naturally-aligned scalars on both ISAs). The sentence reads as if every atomic op uses CAS. Minor framing issue. Confidence: MED (the claim is textually wrong but most readers will map it to "atomics use these when needed"). Not worth blocking on.
